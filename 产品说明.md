# 《卅一景》产品说明

## 产品概述

《卅一景》是一款以明代苏州拙政园为背景的**叙事解谜互动网页游戏**——融合文人美学、园林空间叙事与 AI 对话技术的沉浸式互动体验。

游戏以文徵明《拙政园三十一景图》这套真实传世画作为叙事支点，构建了一个跨越五百年的"观看与被观看"的故事。玩家不是在寻找一幅丢失的画，而是在还原一种被历史体制性地吸收、遮蔽的**观看方式**——这是一个关于"谁有资格被记住"的追问。

---

## 产品定位

### 类型定位

文人悲情 / 双世界解谜 / 园林视角探秘 —— 互动叙事游戏

### 核心体验

玩家扮演古画修复研究生**沈念**，在"现实修复工作室"与"画中拙政园"两个世界之间穿梭，通过修复残画、收集信物、与 AI 导师对话、解读视角线索，逐步还原一位被历史遗忘的女性画者——**王蘅**——留下的低位观看视角。

### 核心主题

"**要被见证，不要被正名**"

王蘅不求历史署名，只希望她所见过的世界仍有人记得。第三十一景的画面保留了她发现的低位视角；文徵明以自己的笔保存了她的眼睛。后人并没有重画此景，而是在重装、配边、归档过程中遮蔽了说明视角来源的边注、题签和残字。玩家要找回的不是一幅失踪的画，而是被正式作品体系吸收掉的观看来源。

### 差异化价值

| 维度 | 《卅一景》的选择 |
|------|------------------|
| 叙事内核 | 不是"破案找凶手"，而是"为一个被遗忘的人完成见证" |
| 解谜逻辑 | 不是机关密码，而是视角还原——"从哪里看、看到什么、为什么被遮蔽" |
| AI 运用 | 不是通用聊天机器人，而是受知识约束的"修复笔记本"，按进度解锁信息 |
| 美学取向 | 不是西式奇幻或日式二次元，而是文人画卷的现代转译 |
| 情感基调 | 不是恐惧或刺激，而是沉静中逐渐涌起的悲悯与确认 |

## 目标用户

### 核心画像

偏好叙事驱动型解谜游戏的**成年中文用户**，对传统文化有亲近感，审美上排斥廉价玄幻和过度商业化。

### 用户特征

- **文学/历史/艺术爱好者**：对明代文人文化、园林美学、古画修复有天然兴趣
- **叙事体验追求者**：期待的不只是"通关"，而是在故事中获得情感共鸣
- **慢节奏偏好**：愿意花时间阅读、观察、思考，不追求快速反应
- **审美敏感度高**：注重视觉品质，排斥粗糙的界面设计和廉价的动效

### 游玩场景

- 单人游玩，安静环境
- 连贯的叙事体验，单次时长 30-60 分钟
- 电脑端为主（需要精细的画面观察和文字阅读）
- 游戏不要求快速反应，强调思考、观察与情感共鸣

### 情感价值

玩家在游戏结束时获得的不是"胜利感"，而是一种安静的确认——"有人看到了"。这是一种罕见的、属于文人美学的游戏情感体验。

---

## 品牌个性

**古雅 · 沉静 · 留白**

- **古雅**：不是复古做旧，而是传统文人审美的现代转译。宣纸质感、水墨晕染、篆刻红印、书法字体，以克制的当代方式呈现。
- **沉静**：界面不抢夺叙事注意力。动画舒缓、颜色内敛、信息密度低。让玩家有时间停下来看、停下来想。
- **留白**：空白不是缺失，是呼吸。文字与界面元素之间留有足够空间，让玩家在阅读叙事文案时不受压迫。

## 游戏章节结构

```
着陆页 → 主菜单（章节选择）
         ├── 序章 · 残页      工作室：初识残画、获得笔记本与物件匣
         ├── 第一章 · 东园    画中：兰雪堂/芙蓉榭/缀云峰 探索解谜
         ├── 第二章 · 中园    画中：远香堂/小飞虹 探索解谜 + 诗词对照
         ├── 第三章 · 西园    画中：留听阁/鸳鸯馆 探索解谜
         └── 终章 · 第三十一景
              ├── 真相空间入口（周鹤年最终提醒）
              ├── 四问复原谜题（核心玩法）
              ├── 第三十一景完成（全画显现）
              ├── 王蘅回声（情感高潮）
              └── 三结局分支
                   ├── 结局一 · 存档（写入修复报告）
                   ├── 结局二 · 守密（有些痕迹被看见就够了）
                   └── 结局三 · 续笔（画一幅新的画）
```

每个章节包含**画中世界**（温暖黄赭色调）与**修复工作室**（冷静灰蓝色调）双场景切换。

## 核心玩法

### 探索与收集

- 画中世界场景内设置可交互光点，点击触发线索发现
- 收集三样关键信物：**断簪**（芙蓉榭）、**残砚**（小飞虹）、**草图拓片**（留听阁）
- 物件匣统一管理收集物，跨章节延续

### AI 对话系统

- 修复笔记本内置 AI 助手（DeepSeek 驱动）
- AI 仅能基于当前进度已解锁的知识片段回答，防止剧透
- "周老师批注"形式做引导，非人格化对话
- 离线降级：API 不可用时自动切换预设对话

### 综合研讨门槛

- 三项线索集齐后解锁"综合研讨"按钮
- 玩家在笔记本面板内进行推理讨论
- 前端关键词累积匹配通过后，结论自动记入笔记本
- 通过后解锁下一章节入口

### 终章四问复原

| 问题 | 形式 | 正确答案 |
|------|------|----------|
| Q1：谁在看？ | 双层选择 | 王蘅 → 被遮蔽的观看者 |
| Q2：从哪里看？ | 地图点击 | 芙蓉榭栏杆下 |
| Q3：看见了什么？ | 多选 | 远香堂倒影/小飞虹弧线/梧竹幽居竹影 |
| Q4：为什么没人知道？ | 单选 | 边注题签被重装遮蔽 |

每答对一问，背景画面对应区域从线稿渐变为水墨着色（朱砂辉光动画）。

## 视觉设计

### 设计语言

- **古雅沉静**：宣纸质感、水墨晕染、篆刻红印、书法字体
- **留白呼吸**：低信息密度，文字与界面元素间保持充足空间
- **双世界色调**：画中世界（暖黄赭 `#2a2218`）/ 修复工作室（冷灰蓝 `#1a1f2e`）

### 转场设计

- 各章节独立过渡页：专属背景图 + 慢速平移动画 + 氛围文字渐入
- 支持 Space / Z 键快进
- 场景间切换使用 opacity crossfade

### 交互反馈

- 按钮：深色朱砂风格或浅色宣纸风格（用于选项区分）
- 高亮：朱砂辉光 + glow 动画
- 竖排中文标注
- 逐字渗入动画（回声场景）

## 技术实现

### 技术架构

| 层级 | 技术选型 |
|------|----------|
| 构建工具 | Vite 8 |
| 前端框架 | 无框架，原生 JavaScript（ES Module） |
| 样式方案 | 原生 CSS（CSS 变量主题系统） |
| AI 接入 | DeepSeek API（流式对话） |
| 数据持久化 | localStorage（存档/进度/对话记录） |
| 字体 | Noto Serif SC + LXGW WenKai（霞鹜文楷） |
| 部署 | 纯静态，任意 HTTP 服务器 |

### 核心引擎模块

| 模块 | 职责 |
|------|------|
| `game-engine.js` | 游戏状态机、全局事件总线、章节进度管理 |
| `scene-manager.js` | 场景注册/切换/生命周期（enter/exit） |
| `save-system.js` | localStorage 存档（自动保存 + 手动存档位） |
| `dialogue.js` | 对话队列、打字机效果、回调控制 |
| `inventory.js` | 物件匣收集物管理 |
| `ai-service.js` | DeepSeek API 封装（流式/非流式） |
| `ai-prompts.js` | 五种 AI 角色 System Prompt 模板 |
| `discussion-gate.js` | 综合研讨门槛（关键词累积匹配） |
| `knowledge-base.js` | 轻量 RAG（按进度解锁知识片段注入 prompt） |
| `fallback-dialogues.js` | 离线降级对话（API 不可用时使用） |

### UI 组件

| 组件 | 功能 |
|------|------|
| `NarrationBar` | 底部叙事对话框（打字机效果/立绘/选项/旁白态） |
| `NotebookFloating` | 右侧悬浮笔记本面板（AI 对话/记录/工具/研讨态） |
| `PaintingViewer` | 古画查看器（缩放/拖拽/滤镜/线索探索/渐进提示） |
| `HudBar` | 右下角 HUD 按钮栏 |
| `InventoryPopup` | 物件匣弹出浮层 |
| `FallTransition` | 跌入画中转场动画 |
| `PoemCompare` | 诗词对照模块 |

### 数据流与存档

```
localStorage
├── save_game          # 主存档（当前章节/场景状态/进度标记）
├── inventory_items    # 物件匣（跨章节累积）
├── notebook_records   # 笔记本记录（AI 对话历史 + 研讨结论）
├── knowledge_unlocked # 已解锁知识片段 ID 列表
└── chapter_flags      # 各章节完成标记
```

数据自序章至终章全程贯通，确保章节切换时数据完整继承。
