# AI 深度参与改造方案——从「可选辅助」到「必需推理伙伴」

## 一、设计目标

将 AI 从当前的「锦上添花辅助层」升级为「玩家推理的必需搭档」，与玩家一起在预设的剧情框架内推理解谜。

### 核心设计原则

1. **AI 参与推理，但不代替推理**：AI 提供分析视角和专业知识，引导玩家思考，但最终判断由玩家做出
2. **预设框架不可突破**：AI 的所有回应都在预写的叙事节拍和谜题答案范围内运作——不生成新剧情，不改变谜底，不创造新线索
3. **协作感真实**：玩家感受到的是"和一个真正懂行的学者一起研究"，而非"和一个答题机器对话"
4. **离线仍可通关**：AI 不可用时，系统自动降级为预写引导对话，保证流程不断

---

## 二、总体架构——「协作推理」模型

```
玩家发现证据 → 触发研讨门槛 → 与 NPC 自由对话 → 系统判断推理合理性 → 推进流程
                                    ↓
                            AI 不可用时 → 预写引导对话轮询
```

### 新增核心概念：「研讨门槛」(Discussion Gate)

在每个章节的关键节点设置**研讨门槛**——玩家必须与 NPC 完成一轮有意义的对话，表达自己对当前线索的理解，才能推进到下一阶段。

**研讨门槛的触发条件**：
- 玩家发现关键线索/完成谜题后
- 章节末返回现实时
- 终章作出结局选择前

**研讨门槛的通过条件**：
- 玩家在对话中**表达出了关键推理概念**（系统通过关键词匹配判定）
- 且对话轮数达到最低要求（防止一句话过关）
- NPC（AI 或预写回复）对玩家的理解给予确认后，玩家主动点击"确认这个推断"完成门槛

> [!IMPORTANT]
> **研讨门槛不是"对话次数"门槛，也不是"选择题"门槛**。玩家需要用自己的话表达理解，系统判断其表述中是否包含关键概念。这更接近于"导师听学生阐述思路后点头认可"，而非"学生勾选正确答案"。

## 三、交互范式：自由对话 + 关键词判定

### 3.1 旧模式 vs 新模式

| 维度 | 旧模式（已废弃） | 新模式 |
|------|-----------------|--------|
| 玩家输入 | 从 ABC 选项中选择 | 自由打字输入自己的想法 |
| 判定方式 | 选对/选错 | 系统关键词匹配判断推理合理性 |
| AI 角色 | 出题人（最后给选择题） | 引导者（全程对话引导） |
| 通过机制 | 选到正确选项 | 表达出关键概念 + NPC 确认 |
| 错误反馈 | "你选错了，再想想" | NPC 针对你的说法给出纠正或追问 |
| 离线降级 | 固定对话树 + 最后选择题 | 固定引导回复轮询 + 关键词判定 |

### 3.2 单轮对话流程

```
┌─────────────────────────────────────────────────────────────┐
│  NPC 开场白 / 引导回复                                         │
│  "你发现装裱接缝有二次揭裱的痕迹。这意味着什么？"                │
├─────────────────────────────────────────────────────────────┤
│  [玩家自由输入]                                                │
│  "我觉得是有人故意把边缘的东西盖住了"                           │
├─────────────────────────────────────────────────────────────┤
│  系统同时做两件事：                                             │
│  ① 关键词判定："盖住" ∈ 关键词库 → 命中"conceal"概念           │
│  ② 发送给 AI 生成自然回复                                      │
├─────────────────────────────────────────────────────────────┤
│  NPC 回复（自然对话）：                                         │
│  "对，你抓住了关键。这不是自然老化，是人为处理。                │
│   但你再想想——为什么要处理边缘？画心不是好好的吗？"            │
├─────────────────────────────────────────────────────────────┤
│  [玩家继续输入]                                                │
│  "边缘可能藏着什么不想让人看到的东西，重装的时候遮掉了"          │
├─────────────────────────────────────────────────────────────┤
│  系统判定：命中"conceal"+"rebind" → 理解度达标                │
│  NPC 回复（确认式）：                                          │
│  "嗯。接缝、压痕、遮盖——这三件事连起来，指向同一个判断。       │
│   你确认这是你的推断吗？"                                      │
├─────────────────────────────────────────────────────────────┤
│  [玩家点击"确认这个推断"] → 门槛通过                          │
└─────────────────────────────────────────────────────────────┘
```

### 3.3 关键词判定机制

每个研讨门槛配置一组**关键概念**，每个概念包含一组同义词/近义词：

```javascript
// gate_prologue_margin（装裱接缝）
concepts: [
  {
    id: 'conceal',
    keywords: ['遮盖', '隐藏', '遮住', '盖住', '掩盖', '遮蔽', '被挡', '被遮'],
    description: '边缘被刻意遮盖'
  },
  {
    id: 'rebind',
    keywords: ['重装', '二次', '揭裱', '重新装裱', '换过', '改装', '后来装'],
    description: '被二次揭裱过'
  },
  {
    id: 'intentional',
    keywords: ['刻意', '故意', '有意', '人为', '不是意外', '不是自然', '有人做'],
    description: '是刻意的而非自然老化'
  }
]
```

**判定规则**：
- 玩家输入中命中至少一个概念的关键词 → 该概念标记为"已理解"
- 已理解的概念数 ≥ `requiredConceptCount`（通常 1-2 个）→ 理解度达标
- 理解度达标 + 对话轮数 ≥ `minRounds` → NPC 给出确认式回复，显示"确认推断"按钮
- 玩家点击确认 → 门槛通过

**容错设计**：
- 玩家可以反复尝试，没有"选错惩罚"
- NPC 会根据玩家的偏离方向给出针对性纠正（AI 模式）或通用引导（离线模式）
- 离线模式下提供"我想到……"快捷按钮，降低输入门槛

## 四、逐章设计

### 序章 · 残页

#### 改造后流程
```
周老师5段预写对话 → 查看古画 → 放大镜探索 → 每找到1条线索 → 【研讨门槛】与周老师讨论线索含义 → 3条线索全部讨论完毕 → 汇聚 → 跌入画中
```

#### 研讨门槛 1 — 装裱接缝残角

**触发**：玩家发现装裱接缝痕迹后自动弹出研讨面板

**系统提示**：*你把放大镜移向装裱边缘，发现了一道不自然的接缝压痕。*

**NPC 开场白**：周鹤年：「你发现了装裱接缝。正常的装裱修复不会在边缘留下这种压痕。」

**关键概念配置**：
| 概念 | 关键词 | 说明 |
|------|--------|------|
| conceal | 遮盖、隐藏、遮住、盖住、掩盖、遮蔽、被挡、被遮、压住、压住了 | 边缘被刻意遮盖 |
| rebind | 重装、二次、揭裱、重新装裱、换过、改装、后来装、裱过 | 被二次处理过 |
| intentional | 刻意、故意、有意、人为、不是意外、不是自然、有人做 | 是人做的，不是自然老化 |

**通过条件**：命中 `conceal` 或 `rebind` 任意一个概念

**NPC 确认式回复（理解度达标后）**：
> 周鹤年点了点头：「你抓住了关键——这不是自然老化，是有人刻意处理过边缘。接缝痕迹说明这一页被二次揭裱过，边缘的某些内容被人为遮盖了。你确认这是你的推断吗？」

---

#### 研讨门槛 2 — "……所见"残字

**NPC 开场白**：周鹤年：「你发现了残字'……所见'。这个字在边注区域，不是画心的一部分。」

**关键概念配置**：
| 概念 | 关键词 | 说明 |
|------|--------|------|
| inscription | 题记、题跋、边注、说明、记录、文字、写了字、题字 | 是一段文字记录 |
| truncated | 裁掉、切掉、截掉、剩下一半、不完整、残片、只剩、漏出 | 被裁切后剩下的残片 |
| viewpoint | 所见、看到、视角、看、观看、视点 | 与"观看"有关 |

**通过条件**：命中 `inscription` + (`truncated` 或 `viewpoint`) 任意一个

**NPC 确认式回复**：
> 周鹤年：「嗯。边注区域的'所见'二字，说明原本有一段完整的题记，记录了某种观看位置。但在重装时被裁掉了大部分，只漏出了边缘的残字。你确认这个判断吗？」

---

#### 研讨门槛 3 — 低位构图辅助线

**NPC 开场白**：周鹤年：「这是一条极淡的构图辅助线。辅助线是打草稿时用来定位的。」

**关键概念配置**：
| 概念 | 关键词 | 说明 |
|------|--------|------|
| low_viewpoint | 低、蹲下、坐下、蹲着、趴着、矮、不高、视线低、位置低、很低 | 观看位置很低 |
| draft_line | 辅助线、起稿线、草稿线、构图线、定位线、铅笔线 | 是画家起稿时画的 |
| unusual | 不寻常、不正常、奇怪、少见、不对、不同、异常 | 这个视角不寻常 |

**通过条件**：命中 `low_viewpoint` + (`draft_line` 或 `unusual`) 任意一个

**NPC 确认式回复**：
> 周鹤年：「对。辅助线意味着画家的观看位置。这条线画得很低——留下这根线的人，当时是蹲下、或者坐在很低的地方看园子的。这不是正常的游观视点。你确认这个推断吗？」

---

> [!NOTE]
> 三个门槛全部通过后，汇聚动画才会触发。这确保玩家在"跌入画中"之前，已经用自己的话表达了基础认知：**这一页的边缘被刻意遮蔽了什么，而且那个"什么"与一个低位视角有关。**

### 第二章 · 中园

#### 研讨门槛 1 — "画非一人"（画中世界 · 笔记本）

**NPC 开场白**：「画。非。一。人。四个字。你觉得是什么意思？」

**关键概念配置**：
| 概念 | 关键词 | 说明 |
|------|--------|------|
| another_person | 另一个人、还有别人、不止一个、别人画的、她画的、合作 | 有另一个人参与 |
| not_alone | 不是一个人、多人、几个人、不只文徵明 | 不是文徵明独立完成 |

**通过条件**：命中 `another_person` 或 `not_alone`

> [!WARNING]
> **设计要点**：此时玩家很可能会理解为"有另一位画家"——这是**正确的阶段性误读**。AI 和离线回复都应**支持**这个理解，不可提前纠正。第二章的引导目标是让玩家建立"有另一个参与者"的认知，至于这个参与者是"画家"还是"观看者"，留到第三章反转。

## 五、AI System Prompt 改造

### 5.1 核心约束修改

**当前约束**：
> AI / 提示系统约束：不生成主线剧情，不决定谜题答案，不改变结局，只根据玩家当前进度提供渐进提示。

**改为**：
> AI 参与约束：
> 1. 不**生成**新的主线剧情——所有叙事节拍预写不变
> 2. 不**直接给出**谜题答案——引导玩家自行发现
> 3. 不**改变**结局——三结局由玩家选择
> 4. **在预设认知节点内引导推理**——每个门槛有明确的"引导目标"
> 5. **配合误读反转节奏**——chapter ≤ 2 时支持"另一位画家"的误读，chapter ≥ 3 时引导认知反转
> 6. **不做通过判定**——系统前端负责判断玩家是否表达出关键概念，AI 只负责自然对话引导


## 六、离线降级方案

### 6.1 设计原则

离线模式的核心体验与在线模式**保持一致**：
- 同样有自由输入框
- 同样有"我想到……"快捷按钮
- NPC 回复是预写的引导语句，轮询展示
- 系统同样做关键词匹配判定

### 6.2 预写引导回复结构

```javascript
const FALLBACK_DIALOGUES = {
  'gate_prologue_margin': {
    title: '装裱接缝',
    opening: '周鹤年：「你发现了装裱接缝。正常的装裱修复不会在边缘留下这种压痕。」',
    // 引导回复池——玩家没说到点子上时轮询展示
    hintPool: [
      '周鹤年：「如果画心没问题，为什么要动装裱？除非……装裱下面藏着什么。」',
      '周鹤年：「后人在重装时，往往为了统一全册体例，而裁切或遮盖边缘不规整的部分。」',
      '周鹤年：「关键不是'有没有处理过'，而是'为什么处理'。被遮盖的是无用残损，还是有人不想让人看到的痕迹？"',
    ],
    // 玩家说对时的肯定回复
    affirmPool: [
      '周鹤年点了点头：「对。这不是自然老化。」',
      '周鹤年：「嗯。你抓住了关键。」',
    ],
    // 理解度达标后的确认式回复
    passResponse: '周鹤年：「你抓住了关键——这不是自然老化，是有人刻意处理过边缘。接缝痕迹说明这一页被二次揭裱过，边缘的某些内容被人为遮盖了。你确认这是你的推断吗？」',
    // 玩家说错方向时的纠正回复
    correctionPool: [
      '周鹤年摇了摇头：「画心本身没有问题。问题在边缘的处理方式上。」',
      '周鹤年：「你再想想。正常的老化不会留下这种规则的压痕。这不是时间问题，是人的问题。」',
    ],
  }
};
```

### 6.3 离线模式对话流程

```
NPC 开场白 → 玩家输入
                ↓
        系统关键词判定
                ↓
    ┌───────────┼───────────┐
    ↓           ↓           ↓
  命中概念    没命中        说反了
    ↓           ↓           ↓
  理解度+1   取 hintPool   取 correctionPool
  取 affirmPool  下一条
    ↓
  理解度达标？
    ↓ 是
  展示 passResponse + "确认推断"按钮
    ↓ 点击确认
  门槛通过
```

### 6.4 快捷输入按钮（离线 & 在线均可用）

在输入框上方显示 3-4 个"我想到……"快捷按钮，降低输入门槛：

```
┌──────────────────────────────────────────┐
│ 我想到…… [边缘被遮住了] [有人重装过] [是自然老化] │
├──────────────────────────────────────────┤
│ [你的推断……                        ] [发送] │
└──────────────────────────────────────────┘
```

点击快捷按钮自动填入输入框并发送。这些按钮的内容来自当前门槛的 `concepts` 关键词的同义表述。

## 十、验证计划

### 在线模式测试
1. 逐章走完全部研讨门槛，验证自由输入 → 关键词判定 → NPC 确认的流程
2. 验证玩家可以用不同的同义表达通过同一个门槛
3. 验证 AI 不会提前透露后续章节内容
4. 验证误读反转节奏——第二章支持误读，第三章引导反转
5. 验证终章结局选择前对话不偏向任何选项

### 离线模式测试
1. 不配置 API Key，验证全部门槛可走通
2. 验证快捷输入按钮正常工作
3. 验证关键词判定在离线模式下同样生效
4. 验证门槛通过后的状态正确保存

### 边界测试
- 玩家输入完全无关内容 → NPC 给出引导，不通过
- 玩家输入部分相关但没说到核心 → NPC 肯定方向但继续追问
- 玩家在研讨门槛中刷新页面 → 从当前门槛重新开始（或最近检查点）
- 玩家反复没说到点子上 → hintPool 循环，不卡死
- 玩家一句话包含多个关键词 → 正常通过

## Open Questions

> [!IMPORTANT]
> **Q1：关键词匹配的精度问题**
> 纯关键词匹配可能有误判（如"没有被遮"会命中"被遮"）。是否需要引入简单的语义否定检测？例如检测"没有""不是""不对"等否定词前缀。

> [!IMPORTANT]
> **Q2：离线模式的快捷按钮内容**
> 快捷按钮的内容如果太接近正确答案，会显得像"提示"。如果太模糊，又起不到降低输入门槛的作用。如何平衡？

> [!IMPORTANT]
> **Q3：路演演示时的 AI 延迟**
> 5 分钟路演中增加了多个 AI 对话门槛。API 延迟可能导致超时。是否需要为路演模式设计"快速通过"机制（如按住 Shift 跳过对话动画）？

> [!IMPORTANT]
> **Q4：笔记本研讨 vs 周鹤年研讨的区分度**
> 画中世界的笔记本研讨和返回现实的周鹤年研讨，在交互上是否足够区分？是否需要不同的视觉风格（笔记本纸张背景 vs 工作室场景）来强化差异？
