# AI 实现方案与后续改进说明

## 一、当前 AI 实现目标

当前版本的 AI 接入服务于“修复笔记本”玩法。玩家不是在与周老师本人实时对话，而是在翻阅周老师预先留下的批注、摘录和方法提示。AI 的职责是辅助玩家梳理已发现线索，不能替玩家生成新剧情、提前透露后续章节或直接给出终章结论。

当前实现遵循以下边界：

1. 所有笔记本回复统一以“周老师批注：”开头。
2. AI 只能基于通用修复知识和当前已解锁剧情知识回答。
3. 玩家尚未发现的线索、后续章节内容、终章真相和未解锁身份信息，不主动透露。
4. 玩家自由输入无法依据已解锁资料回答时，回复“周老师批注：翻了翻，没有找到相关记录。”
5. 游戏预先设计的快捷问题必须得到回应，但可以采用保守分析方式，不直接给出明确结论。

## 二、技术接入方式

当前 AI 接入采用“前端状态传递 + 后端轻量 RAG + Prompt 约束”的方式。

### 1. 后端代理

后端位于 `Web/server/`，提供本地 AI 代理接口：

- `POST /api/notebook/message`：修复笔记本主接口。
- `POST /api/chat`：兼容旧接口。
- `GET /api/health`：健康检查。

模型默认使用 `deepseek-v4-flash`。API Key 保存在服务端 `.env` 中，前端不直接暴露密钥。

### 2. 前端统一调用

前端通过 `Web/src/core/ai-service.js` 调用后端接口。各章节在玩家提问时，会把当前上下文一起传给后端，包括：

- 当前章节。
- 当前世界：现实 / 画中。
- 当前子场景。
- 已发现线索。
- 已获得物件。
- 关键进度变量。
- 快捷问题标记。

这些状态用于判断哪些知识可以进入本轮 prompt。

### 3. 知识库筛选

后端在 `Web/server/index.js` 中维护 `NOTEBOOK_KNOWLEDGE`。每条知识片段都有章节号和解锁条件。例如：

- 序章：题签、旁注、底层细线、二次装裱。
- 第一章：兰雪堂匾额、低处观察、芙蓉榭倒影、断簪与“蘅”。
- 第二章：异文比对、“画非一人”、旧批注、残砚朱砂、水面回声。
- 第三章：散落画纸、渗字、封墙线条、朱砂线、低位视角草图、王蘅的信、匾额追认。

后端只把当前已解锁的知识片段注入模型，避免模型提前接触未发现线索。

## 三、当前章节接入情况

### 序章

序章 AI 重点是帮助玩家理解：

- 旧题签残角可能指向二次装裱。
- “……所见”残字可能是观看记录。
- 底层细线不是裂纹或普通装裱痕。
- 三条线索共同指向“来源说明被遮蔽”。

序章阶段禁止透露王蘅、画中世界、后续章节和终章真相。

### 第一章

第一章 AI 重点是帮助玩家理解：

- 兰雪堂匾额多出的一笔是私人标记，而非败笔。
- 低处观察是一种方法，还不是最终结论。
- 芙蓉榭倒影谜题需要关注现实与倒影之间的错位。
- 断簪上的“蘅”只能暂时理解为私人记号，不能直接确认身份。

第一章阶段禁止确认“蘅”字背后的具体人物。

### 第二章

第二章 AI 已按真实变量进行解锁：

- `poemDiffsFound >= 4` 后解锁“画非一人”。
- `foundOldComment` 后解锁旧批注与规范化遮蔽。
- `hasInkstone` 后解锁残砚与朱砂。
- `heardVoice` 后解锁水面回声。

第二章可以讨论“另有参与者”和材料证据链，但不能确认王蘅完整身份或第三章信件内容。

### 第三章

第三章 AI 已补充完整知识库：

- 鸳鸯馆观察。
- 散落画纸。
- 渗字“看得到吗”。
- 留听阁封墙线条。
- 朱砂线显现。
- 低位视角草图。
- “不是画工”的判断。
- 王蘅的信。
- 匾额追认。

其中 `hasLetter` 出现前，AI 不确认王蘅身份；`hasLetter` 后可以讨论王蘅“要被见证，不要被正名”的动机。

### 终章

终章当前暂不开放 AI 总结报告。笔记本仍以回顾线索为主，不参与玩家最终选择，也不替玩家判断哪一种结局更正确。

## 四、快捷问题处理

各章节会给玩家提供预先设计的快捷问题。此前这些问题和玩家自由输入走同一套拒答逻辑，导致部分设计好的问题被回复“未找到相关记录”。

当前已调整为：

1. 快捷问题会带上 `isPresetQuestion` 标记。
2. 后端识别为预设问题后，不直接拒答。
3. 如果问题涉及未完全解锁的信息，AI 只做方向性分析。
4. 回答会以“目前笔记本中还没有更明确的记录/身份指向/结论记录”收束。

例如：

> 周老师批注：这句话可以先按观看方法理解，重点不在三处景物本身，而在它们如何被同一个低而偏的视线同时组织起来。目前笔记本中还没有更明确的结论记录。

## 五、当前限制

当前版本已经能支持试玩，但仍存在一些需要继续打磨的问题：

1. 部分问题因知识库内容较少或解锁状态较严格，AI 会采用较固定的保守模板。
2. 模板式回答虽然能避免剧透，但有时会显得泛泛，例如：
   “这个问题可以先作为线索关系来分析，重点看它牵涉的是字迹、材料、视角、装裱还是题诗。就目前笔记本记录而言，还没有更明确的结论。”
3. 第三章之后的终章 AI 总结报告尚未完成。
4. 当前知识库主要写在后端代码中，后续如果内容继续增加，可以考虑拆成独立数据文件，便于剧情和谜题组维护。

## 六、后续改进方案

### 1. 针对固定模板回答进行剧情化优化

后续需要结合故事剧情和谜题设计，为每个预设问题补充更具体的引导内容，减少泛化模板。

改进方向：

- 为每个快捷问题建立对应的“推荐回答策略”。
- 回答中优先引用当前章节已出现的具体线索，例如断簪、朱砂线、旧批注、草图拓片等。
- 对涉及后续剧情的问题，不直接拒答，而是明确说明“当前只能从某个线索层面分析”。
- 对同类问题设计不同措辞，避免重复出现同一模板。

目标是让 AI 的回答更像周老师预留在笔记本里的具体批注，而不是通用占位语。

### 2. 增补知识库颗粒度

当前知识库以“章节关键线索”为主，后续可以继续细化到“每个环节、每个快捷问题、每个谜题状态”。

建议补充：

- 每个快捷问题的意图说明。
- 每个谜题阶段可提示的观察方向。
- 每条线索在当前阶段可说到什么程度。
- 不能说的内容和可替代说法。

这样可以减少 AI 因资料不足而进入固定模板。

### 3. 终章 AI 总结报告

终章后续计划加入 AI 总结报告，但需要单独设计边界。

建议方向：

- 玩家完成结局选择后触发，不在选择前触发。
- 报告只基于玩家已完成的证据链和选择立场生成。
- 不评价玩家选择对错。
- 三个结局可以分别生成不同风格的报告：
  - 历史正义：偏修复档案 / 研究报告。
  - 私人守护：偏私人札记 / 保留性说明。
  - 当代续写：偏创作说明 / 展陈文本。

终章报告需要在剧情组确认三结局文本和价值边界后再接入。

## 七、当前给试玩成员的使用说明

试玩时需要同时启动前端和后端：

```bash
cd Web/server
npm install
npm start
```

另开终端：

```bash
cd Web
npm install
npm run dev
```

本地后端需要配置 `Web/server/.env` 中的 `DEEPSEEK_API_KEY`。该文件不应提交到仓库。
