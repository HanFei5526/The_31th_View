# 完整流程与存档解锁一致性走查

## 状态

已完成。

## 修改目标

- 复核从着陆页、菜单、序章、第一章、第二章、第三章到终章的场景注册与跳转是否一致。
- 检查存档恢复、章节解锁、物件继承和进度兜底是否会造成卡关、提前解锁或记录缺失。
- 对照当前实现与 `Web/implementation_plan.md`，修正文档中过期的“占位章节”描述。
- 重点验证断簪、残砚、草图拓片、王蘅的信等关键物件是否能跨章进入终章。
- 检查 AI 面板关闭、后端不可用、刷新页面、从菜单继续游戏等流程是否有可理解的兜底。

## 为什么这一条排在现在

已经完成的前两类高风险问题是：

- AI Key 安全：前端不再暴露真实密钥。
- AI 知识边界：AI 不提前剧透，也不替玩家做终章判断。

下一层风险就是流程稳定性。如果玩家在章节切换、存档恢复或终章入口处卡住，前面的安全和知识边界都会失去实际意义。

## 重点检查项

### 1. 场景注册与菜单入口

- 当前已实现页面是否都被注册。
- 菜单中的“继续游戏”“章节入口”是否指向真实场景。
- 文档中是否还把已经实现的章节写成占位。

### 2. 章节完成与解锁标记

- `prologueComplete`
- `chapter1Complete`
- `chapter2Complete`
- `chapter3Complete`
- `finaleComplete`

检查这些标记是否在正确时机写入，是否会过早或过晚。

### 3. 关键物件继承

- 修复笔记本
- 断簪
- 残砚
- 草图拓片
- 王蘅的信

检查刷新、直接进入后续章节、终章兜底时是否缺物件或重复加入。

### 4. 记录页与 AI 上下文

- 跨章节记录是否保留。
- 旧记录是否会重复。
- 存档恢复后，AI 上下文是否和玩家实际进度一致。

### 5. 后端不可用 / AI 不可用

- 后端未启动或未配置 API Key 时是否走降级文案。
- 玩家是否仍能完成主线。
- AI 面板关闭或不可用时，快捷问题、跳过按钮、主线推进是否正常。

### 6. 终章入口

- 第三章完成后是否稳定进入终章。
- 终章是否能拿到前序关键物件。
- 四问复原与三结局是否能推进到结束卡。

## 主要改法

1. 先做只读代码走查，列出实际问题。
2. 对明确的小问题做最小改动。
3. 不重构场景系统，不改变剧情主线。
4. 若发现较大风险，先汇报再拆成下一条单独处理。
5. 同步更新过期实现文档。

## 完成标准

- 场景注册、菜单入口和测试阶段章节全开逻辑一致。
- 存档恢复不会导致关键物件缺失、重复或章节卡死。
- 从第三章进入终章时，终章所需物件和进度完整。
- AI 不可用时主线仍可推进。
- `Web/implementation_plan.md` 不再把已实现章节描述为占位。
- 通过语法检查和前端构建。

## 验证方式

- 搜索并复核场景注册、章节完成标记、物件继承逻辑。
- 运行 JavaScript 语法检查。
- 运行前端构建。
- 如需要，启动前端做代表性浏览器走查。

## 测试阶段保留项

- 菜单章节入口当前保持全开，便于直接跳转测试第二章、第三章和终章。
- 上线前需切回按完成标记逐章解锁：序章默认开放，序章完成后开放第一章，第一章完成后开放第二章，第二章完成后开放第三章，第三章完成后开放终章。
