# 大场景入口存档点：读档回滚当前段落

## 状态

已实施完成。

## 背景问题

当前存档使用浏览器 `localStorage`，键名为 `sanyijing-save`。它能保存当前场景、进度、物件匣和笔记本记录，但目前更接近“场景级保存 + 关键进度保留”。

这会带来一个问题：

- 玩家在某个大场景内部已经获得线索或物件，但没有完成该大场景。
- 玩家退出、刷新或返回菜单后再继续。
- 数据层面可能保留了中途线索或物件。
- 演出层面却可能从场景入口重新播放。

这会造成“记录里已经有，但场景又像第一次发现”的错位。

## 设计目标

- 不做精确到每句台词、每个热点、每个动画的中途恢复。
- 只在每个“大场景入口”建立正式存档点。
- 大场景内部获得的线索、物件、记录，在该大场景完成前只视为运行时临时进度。
- 玩家读档时回到最近的大场景入口。
- 当前未完成大场景内的临时进度被回滚。
- 已完成大场景之前的成果保留。
- 提示、快捷问题、笔记本展开/收起都按大场景入口重新初始化。

## 核心规则

### 1. 正式存档点只存在于大场景入口

正式 checkpoint 只在大场景开始时写入：

```text
进入大场景入口
→ 生成入口快照
→ 写入 localStorage
```

### 2. 大场景内部不写正式 checkpoint

大场景内部可以继续更新运行时状态，例如：

- 当前场景热点是否点过
- 当前场景线索是否发现
- 当前场景临时物件是否拿到
- 当前轻量讨论是否完成

但这些不作为下次“继续游戏”的正式恢复点。

### 3. 完成大场景后推进到下一个 checkpoint

大场景完成时，才把这一段的成果固化，并进入下一个大场景入口 checkpoint。

```text
完成当前大场景
→ 固化本段成果
→ 进入下一个大场景入口
→ 写入新 checkpoint
```

### 4. 读档只恢复 checkpoint 快照

继续游戏时：

```text
读取 checkpoint
→ 恢复 checkpoint 内 progress / inventory / notebookRecords
→ 切换到 checkpoint.scene
→ 该场景从入口开始重新初始化
```

### 5. 当前段落中途结果回滚

如果玩家在当前大场景中途退出，下次继续：

- 保留 checkpoint 之前已经固化的成果。
- 清除当前大场景内未固化成果。
- 笔记本记录也回滚到 checkpoint 时的记录。
- 物件匣也回滚到 checkpoint 时的物件。
- AI 知识边界随 checkpoint 恢复，不保留当前段落中途知识。

## 与提示 / 笔记本 / 快捷问题的关系

这套机制不保存 UI 中间态。

读档后，大场景入口负责重新设置：

- 系统提示
- 快捷问题
- 笔记本展开或收起
- 轻量讨论状态
- 当前可交互热点

因此：

- 普通探索入口：笔记本默认收起，只更新快捷问题。
- 轻量讨论入口：笔记本自动展开。
- 综合研讨入口：笔记本自动展开并锁定。
- 终章回顾入口：笔记本展开到记录页。

这样可以避免保存“笔记本刚好打开到某个 Tab”“某个提示正在播放”这类脆弱状态。

## 数据结构建议

在 `localStorage` 仍使用 `sanyijing-save`，但扩展字段：

```js
{
  checkpointId: 'chapter2_xiaofeihong_start',
  chapter: 2,
  scene: 'chapter2',
  world: 'paint',
  inventory: [...],
  progress: {...},
  notebookRecords: [...],
  notebookChatsByChapter: {...},
  timestamp: 123456789,
  version: '1.1.0'
}
```

新增一个引擎 API：

```js
saveCheckpoint(checkpointId, snapshot)
```

或由配置表驱动：

```js
this.engine.saveCheckpoint('chapter2_xiaofeihong_start');
```

## 回滚策略

### 推荐做法

保存 checkpoint 时直接保存完整快照：

- `progress`
- `inventory`
- `notebookRecords`
- `notebookChatsByChapter`
- `scene`
- `chapter`
- `world`

读档时完全恢复该快照。

这样不需要手写“哪些 key 要删除”，也避免漏删。

### 不推荐做法

不建议只保存当前进度，再读档时用黑名单删除当前段落 key。

原因：

- 容易漏掉笔记本记录。
- 容易漏掉物件。
- 后续新增线索时必须同步维护黑名单。

## 对现有保存点的处理

当前很多 `saveProgress()` 位于线索或物件获得后。实现 checkpoint 后，需要重新定义它们的作用。

建议：

- `saveProgress()` 保留给“章节完成 / 结局完成 / 手动调试”。
- 大场景内部获得线索时，不写正式存档，只更新运行时状态。
- 大场景完成并切到下一个入口时，调用 `saveCheckpoint()`。

需要逐一复核当前保存点：

- 序章：获得笔记本、综合研讨通过、序章完成
- 第一章：断簪获得、第一章完成
- 第二章：旧批注、水面回声、残砚、第二章完成
- 第三章：散落草图、渗字、红线、草图、拓片、信件、讨论完成、匾额追认、第三章完成
- 终章：结局选择完成

其中，属于当前大场景内部的保存点应改为运行时更新，不写正式 checkpoint。

## 实现步骤建议

### 第一步：只加 checkpoint 框架，不改所有章节逻辑

- 扩展 `SaveSystem` 支持 `checkpointId`。
- 在 `GameEngine` 增加 `saveCheckpoint(checkpointId)`。
- `continueGame` 改为读取 checkpoint 快照。
- 保留现有 `saveProgress()`，避免一次性风险过大。

### 第二步：从第二章或第三章试点

建议先拿第二章试点：

- `chapter2_yuanxiang_start`
- `chapter2_xiaofeihong_start`
- `chapter2_workshop_start`

原因：第二章结构清楚，只有远香堂和小飞虹两个大场景，风险低于第三章。

### 第三步：推广到第一章、第三章、终章

第二章验证稳定后，再逐章接入。

### 第四步：清理旧保存点

确认 checkpoint 稳定后，再把大场景内部的 `saveProgress()` 降级为运行时状态更新。

## 风险点

- 当前各章节场景内部状态并不是为“入口恢复”专门设计的，可能需要补一些入口判断。
- 如果一个场景类包含多个大场景，例如 `chapter3-paint.js`，读档到同一个 scene 后，还需要知道应进入哪个内部子场景。
- 目前 checkpoint 只保存 `scene` 还不够，还需要保存 `subScene` 或让场景根据 `checkpointId` 跳到对应入口。
- 笔记本记录必须和 progress 同步回滚，否则会出现记录和状态不一致。
- 物件匣必须和 progress 同步回滚，否则会出现物件提前存在。

## 完成标准

- 继续游戏回到最近大场景入口，而不是当前段落中途。
- 当前未完成大场景中的线索、物件、记录不会保留到下次继续。
- 已完成大场景之前的成果完整保留。
- 提示、快捷问题、笔记本展开状态由入口重新初始化。
- 第二章试点验证通过后，再推广到其他章节。

## 暂不实施内容

- 不做精确到热点、台词、动画帧的中途恢复。
- 不做服务器存档。
- 不做多存档槽。
- 不做云同步。
- 不改动剧情主线。
